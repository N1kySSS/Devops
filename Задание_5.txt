Практическое задание №5
Проектирование безопасной CI/CD-инфраструктуры с Vault, Docker Registry и Jenkins

Цель работы: 
Необходимо спроектировать и развернуть инфраструктуру для автоматической сборки Docker-образов (для тестирования – сборка приложения из прошлого задания), в которой все секреты (учётные записи, сертификаты) хранятся в HashiCorp Vault, доступ к приватному Docker Registry разграничен, а сборка выполняется на удалённом хосте с взаимной TLS-аутентификацией.

Задание: 
Создать работающий комплекс из Vault (хранилище секретов и PKI), приватного Docker Registry с аутентификацией на основе данных из Vault, удалённого Docker daemon, защищённого mTLS, и Jenkins, выполняющего сборку тестового приложения. Необходимо продемонстрировать разграничение прав пользователей registry (reader – только pull, writer – push/pull) и динамическое получение секретов Jenkins из Vault без их сохранения.

Общие требования: 
Все соединения между компонентами защищены TLS. Vault выступает корневым CA и выпускает сертификаты для Docker daemon, Registry и клиентов (Jenkins) с ограниченным TTL. Vault хранит учётные данные пользователей registry (reader и writer) в защищённом виде.

Требования к Vault
Vault инициализируется с использованием механизма печати (seal) с тремя ключами и порогом в два ключа для распечатывания (кворум 2 из 3) (не dev режим). После завершения начальной настройки – создания корневого CA, политик, ролей и учётных записей пользователей registry – root-токен должен быть отозван (revoked). Вся дальнейшая работа с Vault ведётся через ограниченные токены или методы аутентификации (AppRole, Kubernetes и т.п.), обладающие минимально необходимыми правами. Vault выступает корневым центром сертификации и выпускает сертификаты для всех компонентов с ограниченным TTL, а также хранит учётные данные пользователей registry (reader и writer) в защищённом виде (bcrypt-хэши).

Требования к Docker Registry:
Registry доступен по HTTPS. Аутентификация выполняется на основе данных из Vault любым безопасным способом. Авторизация различает пользователей: reader имеет право только на pull указанного репозитория, writer - на push и pull.

Требования к удалённому Docker daemon:
Демон Docker слушает TCP-порт 2376 с обязательной проверкой клиентских сертификатов (mTLS). Серверный сертификат выпущен Vault. Jenkins подключается, предъявляя клиентский сертификат, полученный из Vault во время сборки.

Требования к Jenkins:
Jenkins аутентифицируется в Vault с минимальными правами (чтение клиентских сертификатов и учётных данных writer). Учётные данные для доступа к Vault (токен, RoleID/SecretID и т.п.) хранятся в Jenkins защищённо, никогда не прописываются в открытом виде в Jenkinsfile или конфигах. В пайплайне секреты запрашиваются из Vault, используются для подключения к Docker daemon и registry. При сборке образа необходимо использовать механизмы кеширования, чтобы ускорить повторные сборки. Логи не должны содержать незамаскированные секреты.
